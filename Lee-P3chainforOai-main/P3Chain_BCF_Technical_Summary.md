## 1. 项目定位与背景

`p3-chain` 在 `Lee-P3chainforOai` 项目中的角色，本质上是一个面向 5G 核心网控制平面的区块链功能模块（Blockchain Capability Function, BCF）。其目标不是替代 5G 核心网原有控制逻辑，而是在零信任环境下，为 OpenAirInterface（OAI）一类 5G 核心网组件提供可信增强能力。

从系统定位上看，该模块主要围绕以下三类能力展开：

- 为网元提供去中心化身份（Decentralized Identifier, DID）注册、解析和公钥管理能力；
- 为关键控制面操作提供链上审计存证能力，满足不可篡改（Integrity）与不可抵赖（Non-repudiation）要求；
- 为网元间访问和敏感接口调用提供基于 DID、公钥和令牌的安全鉴权能力。

因此，BCF 在整个系统中的定位可以概括为：

> 面向 5G 零信任控制面的“可信身份 + 审计存证 + 安全鉴权”区块链基础设施。

## 2. 核心功能说明

### 2.1 网元全生命周期管理

项目支持对 5G 网元实例（如 `AMF`、`AUSF`、`SMF` 等）进行完整的生命周期管理，核心接口包括：

- `RegisterNF`
- `GetNFInstance`
- `DeregisterNF`

其实现特征如下：

#### 2.1.1 注册（RegisterNF）

网元注册时，系统接收包含以下信息的请求：

- 网元实例标识 `nfInstanceId`
- DID 标识
- DID Document
- 网元类型 `nfType`
- 网元状态 `nfStatus`
- 可选业务属性，如 `AMFInfo`、`SMFInfo`、地址信息和服务能力信息

注册逻辑会将这些信息封装为链上状态对象，并调用链码中的 `SetNFProfile` 方法写入区块链世界状态（World State）。系统同时构建按 `nfType` 的索引，使后续发现（Discovery）和查询可以按网元类型快速定位实例。

#### 2.1.2 状态查询

查询时，BCF 通过区块链客户端调用 `GetNFProfile` 或 `DiscoverNFByType`，返回指定 DID 对应的 NF Profile 或指定类型的网元列表。  
这意味着控制平面中的“网元可发现性”不再依赖单一中心数据库，而是依赖链上共享状态。

#### 2.1.3 安全注销（DeregisterNF）

网元注销并不是简单删除记录，而是通过链码将 `nfStatus` 更新为 `DEREGISTERED`。  
这种设计具备两个好处：

- 保留历史状态变迁，便于审计和溯源；
- 避免直接删除导致状态丢失，从而提升可追踪性。

换言之，BCF 对网元对象采用的是“可追踪状态演进”模型，而不是“不可恢复的物理删除”模型。

### 2.2 去中心化身份（DID）系统

该项目实现了面向网元的 DID 管理机制，其核心思想是把传统集中式 PKI 中的“身份绑定、公钥分发、信任建立”过程，向链上共享身份状态迁移。

#### 2.2.1 DID Document 的作用

每个网元注册时都会带上一个 DID Document，至少包含：

- `id`
- `verificationMethod`
- `controller`
- `blockchainAccountId` 或公钥材料

这使得 DID Document 既承担“身份声明”的作用，也承担“分布式公钥目录”的作用。

#### 2.2.2 对传统 PKI 的分布式演进

传统 PKI 通常依赖中心化 CA 或集中式目录服务。而在本项目中：

- 身份记录不保存在单点数据库中；
- 公钥绑定关系随 DID Document 上链；
- 多节点共享身份状态；
- 网元可基于链上 DID Document 验证挑战签名、提取链上地址、公钥或验证方法。

因此，这套 DID 机制可以被理解为对传统 PKI 的一种分布式演进：

- “证书目录”由区块链世界状态承载；
- “身份可信性”由链上不可篡改存证保障；
- “主体不可抵赖性”由 DID 与签名关联保证。

#### 2.2.3 在 5G 控制面中的意义

在零信任环境下，网元之间不应默认互信。DID 系统带来的价值是：

- 任一网元都可以先验证对方 DID 是否已注册；
- 再验证 DID Document 中的公钥或链上账户绑定关系；
- 最后通过 challenge-response 或 token 完成鉴权。

这使 5G 核心网内部的服务交互具备可验证身份基础，而不是依赖“内网天然可信”的假设。

### 2.3 链上审计系统（On-chain Auditing）

链上审计系统是本项目最关键的增强能力之一。它的设计目标是：

- 对关键业务操作生成不可篡改的审计记录；
- 记录操作主体、操作对象、结果、时间、请求摘要等关键元数据；
- 审计写链异步化，不阻塞主业务；
- 敏感信息不上链，只保留摘要和最小必要元数据。

#### 2.3.1 审计覆盖范围

当前已接入的重点操作包括：

- `NF_REGISTER`
- `NF_DEREGISTER`
- `DISCOVERY`
- `SUBSCRIPTION_CREATE`
- `SUBSCRIPTION_QUERY`
- `SUBSCRIPTION_DELETE`
- `AUTH_VERIFY`

#### 2.3.2 审计中间件

审计中间件位于请求入口处，主要负责：

- 生成 `trace_id`
- 缓存原始请求体
- 注入请求开始时间

这样后续业务 Handler 在提交审计事件时，就可以自动带上：

- `trace_id`
- `method`
- `resource_path`
- `timestamp`
- `request_hash`

它的作用不是决定“记什么审计”，而是负责提供一套统一的上下文采集机制。

#### 2.3.3 业务 Handler 显式提交审计语义

具体业务 Handler 负责把通用请求上下文转换为业务语义明确的审计事件。例如：

- 注册网元时，提交 `NF_REGISTER`
- 注销网元时，提交 `NF_DEREGISTER`
- 鉴权成功时，提交 `AUTH_VERIFY`

这一步会明确填写：

- `operator_did`
- `operation_type`
- `target_object_id`
- `result`
- `result_code`
- `related_tx_hash`
- `metadata`

换言之，Handler 决定“审计语义”，中间件提供“通用上下文”，两者共同组成完整审计记录。

#### 2.3.4 异步 Worker 机制

审计服务采用典型的 `Asynchronous worker` 架构：

- 主业务线程只负责把 `AuditEvent` 入队；
- `AuditService` 内部维护缓冲队列；
- 多个 worker goroutine 在后台消费队列；
- worker 将事件转换为 `AuditLog`，并通过 `SoftInvoke(CreateAuditLog)` 写链；
- 写链失败时按策略重试，重试耗尽只记录告警，不拖垮主业务。

这套设计使审计系统具备“低侵入、高解耦”的工程特征，尤其适合 5G 控制面这种对时延敏感的业务路径。

#### 2.3.5 请求摘要提取

审计系统不会把原始请求正文完整上链，而是计算一个摘要 `request_hash`。  
摘要输入通常包括：

- HTTP Method
- Request Path
- Operator DID
- Target Object ID
- Canonical Query
- Sanitized Body

摘要算法使用：

- `SHA-256`

输出形式：

- 十六进制编码字符串

这样做的价值是：

- 保证后续可以验证“这次请求对应的正文摘要就是这一个”；
- 同时避免把敏感原文直接暴露到链上。

#### 2.3.6 敏感数据脱敏逻辑

在计算摘要前，系统会对请求体进行脱敏清洗，当前重点过滤的字段包括：

- `authorization`
- `token`
- `signature`
- `nonce`
- `challenge`
- `didDocument`
- `publicKey`

这意味着：

- 敏感原文不会直接写入审计记录；
- 链上保留的是清洗后的摘要和必要元数据；
- 审计系统在满足可审计性的同时，尽量降低敏感数据暴露风险。

## 3. 系统架构详细解析

### 3.1 总体架构

从后端架构角度看，BCF 可以分为三层：

- 控制平面交互层：GIN RESTful API
- 逻辑处理层：Handler、AuditService、DperService
- 底层账本层：p3Chain Core、LevelDB、智能合约引擎

### 3.2 控制平面交互层

控制平面交互层基于 `GIN` 框架实现，主要承担：

- 对外暴露 RESTful API
- 参数校验与 JSON 绑定
- 中间件接入
- 请求上下文管理
- 最终响应封装

这层在项目中的典型表现是：

- `/nbcf_management/v1`
- `/nbcf_discovery/v1`
- `/nbcf_auth/v1`

它本质上是 BCF 与 OAI 5G 核心网或其他 NF 的控制面集成入口。

### 3.3 逻辑处理层

逻辑处理层是整个系统的“业务中枢”，主要包含两个关键组件。

#### 3.3.1 AuditService

`AuditService` 的职责：

- 创建与管理审计队列
- 启动 worker 协程池
- 提供 `Enqueue`
- 后台异步调用链码写入审计日志
- 提供审计查询聚合逻辑

它是系统中典型的“异步后台服务组件”。

#### 3.3.2 DperService

`DperService` 是上层业务与 p3Chain 客户端命令能力之间的桥梁，负责：

- 调用 `SoftInvoke`
- 调用 `SoftCall`
- 屏蔽底层命令行式区块链客户端接口复杂度
- 提供统一的区块链访问门面

可以把它理解为：

> 面向业务逻辑的 Blockchain Gateway / Adapter。

### 3.4 底层账本层

底层账本层由 p3Chain 自身区块链能力提供，主要包括：

- 区块链核心执行与共识框架
- 智能合约执行引擎
- 世界状态（World State）
- 区块数据持久化

#### 3.4.1 p3Chain 区块链核心

p3Chain 是一个联盟链框架，具备：

- P2P 网络能力
- 分层/分片式网络组织能力
- BFT 导向的一致性机制
- 交易池与区块管理能力

#### 3.4.2 LevelDB 状态存储

项目当前主要使用 `LevelDB` 作为底层 K-V 持久化引擎，用于：

- 存储世界状态
- 存储区块及相关索引

选择 LevelDB 的好处在于：

- 读写性能高
- 结构简单
- 易于与链上 K-V 状态模型对接

#### 3.4.3 智能合约引擎

BCF 的 DID、订阅、审计等数据并不是直接写数据库，而是通过智能合约函数进行写入和查询，例如：

- `SetNFProfile`
- `GetNFProfile`
- `CreateSubscription`
- `GetSubscriptions`
- `CreateAuditLog`
- `GetAuditLogByID`

因此，业务状态与审计状态都统一纳入链上合约和世界状态管理体系中。

## 4. 关键技术特性

### 4.1 高性能设计

在 5G 控制面场景中，性能设计的核心矛盾在于：

- 既要留下可信审计痕迹；
- 又不能让每一个控制面接口因为“同步写链”而明显增大时延。

本项目主要通过以下方式保证高性能：

#### 4.1.1 审计异步化

业务请求成功后，审计只做：

- 构造事件
- 入队

而不等待区块链审计交易同步确认。  
这种模式显著减少了主线程等待时间。

#### 4.1.2 Buffered Queue

审计服务内部采用带缓冲的队列，可以吸收瞬时流量突发，避免每次请求都立即阻塞在写链路径上。

#### 4.1.3 Worker 并发消费

通过多个 worker goroutine 并发处理队列中的审计事件，形成“生产者-消费者”解耦结构，提升整体吞吐能力。

#### 4.1.4 SoftInvoke 模式

业务和审计侧均大量使用 `SoftInvoke`。在本项目语义下，`SoftInvoke` 可以理解为一种轻量级的区块链提交模式：

- 发起链上调用
- 不强制要求调用方同步等待完整交易检查结果
- 以较低延迟换取较高控制面响应效率

对于毫秒级时延敏感的 5G 控制面来说，这是一种务实的架构折中。

### 4.2 安全性设计

#### 4.2.1 请求摘要与 SHA-256

项目通过 `SHA-256` 对规范化请求材料做摘要，保留：

- 请求存在性证明
- 请求内容一致性证明

而不直接保留原始敏感内容。  
这是典型的“可证明但不过度暴露”的安全设计。

#### 4.2.2 Non-repudiation（不可抵赖性）

系统的不可抵赖性主要由以下要素共同构成：

- DID 绑定身份主体
- 业务操作与 `operator_did` 关联
- 链上审计记录不可随意修改
- `request_hash` 可用于证明请求内容摘要
- `tx_hash` 和 `related_tx_hash` 建立业务交易与审计交易关联

因此，当某次关键操作发生后，系统可以较强地回答：

- 谁做了这件事；
- 做了什么；
- 对谁做的；
- 在什么时间做的；
- 这条记录对应哪一笔链上事务。

#### 4.2.3 脱敏与最小暴露原则

通过对 `signature`、`nonce`、`token` 等字段的过滤，系统避免把认证秘密、访问令牌等敏感数据直接上链。  
这符合后端安全设计中的最小暴露原则（Least Exposure Principle）。

#### 4.2.4 查询权限控制

审计查询并不是无条件开放的，系统定义了 `audit_read` 权限，并可通过统一的 Token 鉴权中间件控制访问。  
这意味着“谁可以写审计”和“谁可以读审计”可以纳入统一的安全策略域。

## 5. 价值与适用场景

从工程应用视角看，这套 BCF 机制特别适合以下场景：

- 5G 核心网内部多网元零信任互联
- 网元身份需要可验证、可追踪
- 关键操作需要符合审计合规要求
- 不希望依赖单点中心数据库保存关键信任状态

它尤其适用于以下需求：

- 面向高安全等级场景的 5GC 控制面增强
- 面向跨域、跨运营实体协作的可信服务编排
- 面向审计、监管、事故追踪的链上取证体系

## 6. 总结

基于 `Lee-P3chainforOai` 的 p3-chain BCF 模块，并不是一个“简单把区块链接到 5G 上”的实验性拼接，而是一套围绕 5G 控制面零信任需求设计的可信增强体系。

它的技术主线非常清晰：

- 用 DID 建立网元身份信任基础；
- 用链上审计实现关键操作不可篡改与不可抵赖；
- 用异步 Worker 和 `SoftInvoke` 平衡性能与可信存证需求；
- 用脱敏摘要和权限控制保障审计数据安全。

从后端架构角度看，这是一种典型的“控制面业务 + 区块链能力 + 安全中间件”深度融合设计，其价值不在于把所有逻辑都上链，而在于把最关键的身份、信任和留痕能力上链，从而为 5G 核心网提供可验证、可审计、可追溯的可信基础设施。
